# 方案 1：自动执行重构说明

## 重构概览

采用**方案 1（自动执行）**，简化调用链，消除循环依赖。

### 重构前（有问题的设计）

```
OtaClient.handleJobNotification()
  ↓
listener.onNewPackage(pkg)          ← 回调到 OtaDemo
  ↓
client.startUpgrade(pkg)            ← OtaDemo 手动调回 OtaClient ❌
  ↓
listener.onInstallFirmware()        ← OtaClient 再回调 OtaDemo
```

**问题：**
- ❌ 循环依赖：OtaClient → listener → OtaClient → listener
- ❌ `onNewPackage` 变成了"必须调用 startUpgrade"的地方
- ❌ 违背回调语义（回调应该是通知，不是命令）
- ❌ listener 职责不清（既接收通知，又控制流程）

---

### 重构后（清晰的设计）

```
OtaClient.handleJobNotification()
  ↓
listener.onNewPackage(pkg)          ← 仅通知，无需任何操作
  ↓
OtaClient.startUpgrade(pkg)         ← 自动执行，无需 listener 调用
  ↓
listener.onInstallFirmware()        ← 执行安装
```

**优点：**
- ✅ 调用链单向流动，无循环依赖
- ✅ listener 职责清晰（提供信息 + 执行操作）
- ✅ OtaClient 职责清晰（流程控制）
- ✅ 回调语义正确（通知，不是命令）

---

## 核心改动

### 1. OtaClient.java

#### 改动 1：startUpgrade() 改为 private

**重构前：**
```java
public void startUpgrade(OtaPackage pkg) {
    // 公开方法，外部可调用
}
```

**重构后：**
```java
private void startUpgrade(OtaPackage pkg) {
    // 私有方法，仅内部调用
}
```

**原因：** startUpgrade 不应该被外部调用，它由 OtaClient 内部自动触发

---

#### 改动 2：handleJobNotification() 自动调用 startUpgrade()

**重构前：**
```java
private void handleJobNotification(MqttMessage message) {
    // ... 解析 Job Document

    OtaPackage pkg = new OtaPackage(...);
    listener.onNewPackage(pkg);

    // ❌ 等待 listener 调用 client.startUpgrade(pkg)
}
```

**重构后：**
```java
private void handleJobNotification(MqttMessage message) {
    // ... 解析 Job Document

    // 版本检查
    if (version.equals(currentVersion)) {
        reportJobStatus(jobId, "SUCCEEDED", "already running target version");
        return;
    }

    // 校验 Job Document
    if (packageUrl.isEmpty()) {
        reportJobStatus(jobId, "REJECTED", "invalid job document");
        return;
    }

    OtaPackage pkg = new OtaPackage(...);

    // 通知 listener（仅用于日志/UI 更新）
    listener.onNewPackage(pkg);

    // ✅ 自动开始升级
    startUpgrade(pkg);
}
```

**关键变化：**
- 添加了版本检查和 Job Document 校验
- 调用 `listener.onNewPackage(pkg)` 仅用于通知
- 自动调用 `startUpgrade(pkg)`，无需等待 listener

---

### 2. OtaDemo.java

#### 改动：移除手动调用 startUpgrade()

**重构前：**
```java
@Override
public void onNewPackage(OtaPackage pkg) {
    System.out.println("[Demo] New firmware available: " + pkg);
    System.out.println("[Demo] Starting upgrade to version " + pkg.getVersion() + "...");

    // ❌ 手动调用 client.startUpgrade()
    client.startUpgrade(pkg);
}
```

**重构后：**
```java
@Override
public void onNewPackage(OtaPackage pkg) {
    // 仅用于日志输出，OtaClient 会自动开始升级
    System.out.println("[Demo] New firmware available: " + pkg);
    System.out.println("[Demo] OtaClient will automatically start the upgrade...");

    // ✅ 不需要手动调用 client.startUpgrade(pkg)
    // OtaClient 会自动处理
}
```

**关键变化：**
- 移除了 `client.startUpgrade(pkg)` 调用
- `onNewPackage` 只用于日志输出或 UI 更新
- 不需要访问 `client` 实例

---

### 3. OtaListener.java

**无需修改！** 接口保持不变：

```java
public interface OtaListener {
    String onQueryVersion();
    void onNewPackage(OtaPackage pkg);  // 仍然是 void，仅通知
    boolean onInstallFirmware(Path firmwarePath);
    void onProgress(int percent, String description);
}
```

**优点：** 向后兼容，不破坏现有实现

## 调用时序对比

### 重构前（循环依赖）

```
┌──────────────┐
│ OtaClient    │
└──────┬───────┘
       │ handleJobNotification()
       │
       ├─ 解析 Job Document
       │
       ├─ listener.onNewPackage(pkg)
       │      │
       │      ▼
       │  ┌────────────────┐
       │  │ OtaDemo        │
       │  │ onNewPackage() │
       │  └────────┬───────┘
       │           │
       │           ├─ 打印日志
       │           │
       │           └─ client.startUpgrade(pkg)  ❌ 调回 OtaClient
       │                    │
       ▼◄───────────────────┘
   startUpgrade()
       │
       ├─ 下载固件
       │
       ├─ listener.onInstallFirmware()
       │      │
       │      ▼
       │  ┌────────────────────┐
       │  │ OtaDemo            │
       │  │ onInstallFirmware()│
       │  └────────────────────┘
       │
       └─ 上报成功
```

**问题：** OtaClient → OtaDemo → OtaClient，形成循环

---

### 重构后（单向流动）

```
┌──────────────┐
│ OtaClient    │
└──────┬───────┘
       │ handleJobNotification()
       │
       ├─ 解析 Job Document
       │
       ├─ 版本检查
       │
       ├─ 校验 Job Document
       │
       ├─ listener.onNewPackage(pkg)  ✅ 仅通知
       │      │
       │      ▼
       │  ┌────────────────┐
       │  │ OtaDemo        │
       │  │ onNewPackage() │
       │  └────────────────┘
       │      │
       │      └─ 打印日志（不调回 OtaClient）
       │
       ├─ startUpgrade(pkg)  ✅ 自动执行
       │
       ├─ 下载固件
       │
       ├─ listener.onInstallFirmware()  ✅ 执行安装
       │      │
       │      ▼
       │  ┌────────────────────┐
       │  │ OtaDemo            │
       │  │ onInstallFirmware()│
       │  └────────────────────┘
       │
       └─ 上报成功
```

**优点：** 单向流动，清晰简洁

## 使用场景

### 场景 1：正常 OTA 升级

```
[OtaClient] New OTA job received: version=1.0.3
[Demo] New firmware available: OtaPackage{version=1.0.3, ...}
[Demo] OtaClient will automatically start the upgrade...
[OtaClient] Job status -> IN_PROGRESS (downloading firmware v1.0.3)
[Demo] Progress: 10% - downloading
[Demo] Progress: 50% - downloading
[Demo] Progress: 100% - downloading
[OtaClient] Job status -> IN_PROGRESS (verifying firmware checksum sha256)
[OtaClient] Job status -> IN_PROGRESS (installing firmware v1.0.3)
[Demo] Installing firmware: downloads/firmware-1.0.3.bin
[Demo] Verifying firmware signature...
[Demo] Backing up current firmware...
[Demo] Installing new firmware...
[Demo] Firmware installation completed.
[OtaClient] Job status -> SUCCEEDED (firmware v1.0.3 installed successfully)
[OtaClient] Shadow updated: firmwareVersion=1.0.3
```

**注意：** `onNewPackage` 只打印日志，不调用任何方法

---

### 场景 2：版本检查（自动跳过）

```
[OtaClient] New OTA job received: version=1.0.2
[OtaClient] Already running target version 1.0.2, marking job as succeeded
[OtaClient] Job status -> SUCCEEDED (already running target version 1.0.2)
```

**注意：** `onNewPackage` 根本不会被调用，因为版本检查在之前就完成了

---

### 场景 3：Job Document 校验失败

```
[OtaClient] New OTA job received: version=1.0.3
[OtaClient] Invalid job document: missing checksum
[OtaClient] Job status -> REJECTED (invalid job document: missing checksum for security)
```

**注意：** `onNewPackage` 也不会被调用

## 设计优势

### 1. 职责清晰

| 组件 | 职责 |
|------|------|
| **OtaClient** | 流程控制、状态管理、自动执行 |
| **OtaListener** | 提供信息（版本）+ 执行操作（安装）+ 接收通知 |
| **OtaDemo** | 实现 listener 接口，处理业务逻辑 |

### 2. 单向依赖

```
OtaClient ─────> OtaListener <────── OtaDemo
   (调用)          (接口)           (实现)
```

**无循环！**

### 3. 易于测试

```java
@Test
public void testAutoUpgrade() {
    OtaListener mockListener = mock(OtaListener.class);
    when(mockListener.onQueryVersion()).thenReturn("1.0.2");
    when(mockListener.onInstallFirmware(any())).thenReturn(true);

    OtaClient client = new OtaClient("device-001", mockListener, Paths.get("/tmp"));

    // 模拟收到 Job 通知
    // ...

    // 验证
    verify(mockListener, times(1)).onNewPackage(any());  // 通知一次
    verify(mockListener, times(1)).onInstallFirmware(any());  // 安装一次
    // 不需要验证 client.startUpgrade() 被调用，因为它是私有的
}
```

### 4. 易于扩展

如果将来需要添加决策逻辑，可以扩展接口而不破坏现有实现：

```java
// 方案 1 扩展版本（可选）
public interface OtaListener {
    String onQueryVersion();
    void onNewPackage(OtaPackage pkg);

    // 新增：决策回调（可选实现）
    default boolean shouldAcceptUpgrade(OtaPackage pkg) {
        return true;  // 默认接受
    }

    boolean onInstallFirmware(Path firmwarePath);
    void onProgress(int percent, String description);
}
```

## 总结

### 重构前后对比

| 方面 | 重构前 | 重构后 |
|------|--------|--------|
| **调用链** | 循环依赖 | 单向流动 |
| **listener 职责** | 通知 + 控制流程 | 仅通知 |
| **OtaClient 职责** | 被动等待 listener 调用 | 主动控制流程 |
| **startUpgrade()** | public | private |
| **易用性** | 必须手动调用 | 自动执行 |
| **易测试性** | 需要 mock client | 只需 mock listener |

### 关键优势

- ✅ **职责清晰**：OtaClient 控制流程，listener 提供信息
- ✅ **单向依赖**：无循环，易于理解和维护
- ✅ **自动执行**：收到 Job 自动开始，无需手动调用
- ✅ **向后兼容**：不修改 OtaListener 接口
- ✅ **符合惯例**：与华为等主流 IoT SDK 设计一致

### 代码改动量

- **OtaClient.java**: 3 处改动（startUpgrade 改 private + 自动调用）
- **OtaDemo.java**: 1 处改动（移除 client.startUpgrade 调用）
- **OtaListener.java**: 无改动 ✅

**总计：4 行代码改动，架构大幅改善！**
